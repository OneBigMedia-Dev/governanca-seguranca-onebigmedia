Como usar este material.

Este material existe para tirar o atrito técnico antes da Aula 1. Para chegar pronto, cada participante precisa concluir os 3 cursos obrigatórios, instalar todas as ferramentas e mapear um fluxo real de processo com gravação e transcrição.

1

Faça os 3 cursos obrigatórios

Conclua Claude 101, Claude Code 101 e Claude Platform 101 antes da Aula 1. Eles criam a base mínima para acompanhar o programa.

2

Instale todas as ferramentas

Instale e valide VS Code, Git, Node.js, Python, Claude Code CLI e extensão no VS Code. Planeje cerca de 1h30 para instalação e check final.

3

Grave e transcreva um processo

Entre em uma reunião, grave com transcrição e mapeie um fluxo real em detalhe: objetivo, etapas, responsáveis, ferramentas, inputs, outputs, exceções e dores. Avise os participantes e não registre credenciais ou dados sensíveis desnecessários.

4

Peça ajuda cedo

Se travar em acesso, terminal, instalação ou login, acione o time antes da Aula 1. Não instale ferramenta paga, conector ou extensão fora do fluxo oficial.

~/ai-first/fundamentos --operação

Fundamentos e modo de operação.

O programa não exige que todo mundo vire programador. Ele exige clareza para orientar a IA, revisar resultado e saber quando uma automação pode ou não seguir adiante.

LLM sem mistério

Claude gera respostas a partir de padrões e contexto. Ele raciocina bem, mas pode errar com confiança. Por isso revisão humana faz parte do fluxo.

Prompt bom

Inclui contexto, objetivo, restrições, formato desejado, exemplos e critério de sucesso. Quanto mais específico o pedido, menor o retrabalho.

Claude Code

É o Claude com acesso ao projeto: lê arquivos, propõe plano, edita código, roda comandos aprovados e explica o que mudou.

Plan Mode

Use quando ainda não quer editar nada. Primeiro o agente explora, entende a pasta, propõe etapas e só depois executa.

Review

Antes de aceitar resultado, revise arquivos alterados, comandos executados, dados usados, diffs e próximos passos.

Coworker

Trate Claude como par de trabalho: peça alternativas, mande explicar decisões, interrompa caminhos ruins e valide saídas com critério.

Memória do projeto

Use README, docs e, quando fizer sentido, CLAUDE.md para registrar padrões, comandos, decisões e limites recorrentes.

Dados com contexto

Explique origem, permissão, máscara, qualidade e limite de uso. Dado real sensível não entra em prompt sem aprovação.

Humano decide

A IA acelera o trabalho, mas a responsabilidade técnica e de negócio continua com quem constrói e valida o MVP.

~/ai-first/governança --segurança --dados

Governança, segurança e uso de IA.

Todo MVP precisa nascer com limites claros. A regra é simples: menor privilégio, revisão humana, rastreabilidade e nenhum segredo exposto para IA.

Quem é quem: a validação de projetos e a aprovação para produção passam pelo Paulo. O acesso ao Claude Code é liberado pelo Fred Valente. Demandas relevantes devem nascer ou ser registradas no Jira.

Princípios obrigatórios

  • Menor privilégio por padrão.
  • Revisão humana antes de mudança crítica.
  • Segredos nunca ficam no código, prompt, print ou log.
  • Projeto crítico precisa de Jira, PR e registro de decisão.

Nunca envie para IA

  • Senhas, tokens, API keys e service accounts.
  • Arquivos .env, client_secret, .pem ou .key.
  • Dumps de banco e dados de clientes, parceiros ou financeiro.
  • Logs com cookies, credenciais ou dados pessoais.

Pode usar sem aprovação extra

  • Código sem segredo.
  • Logs anonimizados.
  • Dados fictícios, minimizados ou mascarados.
  • Documentação técnica sem informação sensível.

Claude Code

  • Use conta individual liberada pela empresa.
  • Revise comandos antes de executar.
  • Sem autoexecução irrestrita em repositórios críticos.
  • Projeto crítico pede sessão separada e registro.

Ferramentas e custo

  • Não adote SaaS, extensão, conector ou MCP sem aprovação.
  • Todo uso relevante precisa de dono, finalidade e limite de custo.
  • Repositórios oficiais ficam na organização aprovada.
  • Produção exige pipeline de validação.

Banco de dados

  • Preferir treino com SQLite, CSV ou base mascarada.
  • Banco real apenas com aprovação e menor privilégio.
  • Credencial corporativa nunca entra em prompt.
  • Consulta e resultado precisam ser documentados.
Risco baixo

Sem dados sensíveis, sem credenciais e sem produção. Ex.: script local, prova de conceito, documentação.

Risco médio

Usa APIs externas, tokens limitados ou dados internos. Ex.: automação em teste, pipeline em homologação.

Risco alto

Toca produção, banco real, cliente, financeiro ou permissão administrativa. Exige aprovação reforçada.

~/ai-first/desafios --por-área

Cada pessoa traz uma dor real da área.

A escolha do desafio é o ponto mais importante do programa. O melhor caso não é o mais complexo: é aquele que tem impacto claro, dados acessíveis, risco controlado e alguém disposto a validar o resultado.

Como vamos avaliar cada desafio

  • Repetição: acontece toda semana, todo mês ou consome muitas horas?
  • Impacto: reduz erro, economiza tempo, melhora receita, protege informação ou acelera decisão?
  • Dados: os insumos existem e podem ser usados com segurança?
  • Escopo: cabe no ciclo de 8 encontros principais?
  • Risco: envolve produção, dados pessoais, financeiro, clientes, CMS, GCP, GitHub ou credenciais?
  • Dono: existe alguém da área que valide se a solução realmente funciona?
Área e responsávelQuem traz o desafio e quem valida o resultado.
Processo atualComo é feito hoje, com quais ferramentas e quanto tempo consome.
Dor principalErro, retrabalho, atraso, custo, risco ou oportunidade perdida.
Dados envolvidosPlanilhas, arquivos, APIs, bancos, relatórios, e-mails ou sistemas.
Resultado esperadoO que precisa existir ao final para considerar o MVP útil.
Limites de segurançaO que não pode ser exposto, automatizado ou acessado sem aprovação.
