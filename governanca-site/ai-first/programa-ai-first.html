O pre-curso garante que todo mundo chegue na Semana 1 com ambiente completo: VS Code, Claude Code CLI, extensão do Claude no VS Code, Git, Node.js, Next.js via create-next-app, Python, login corporativo e primeira pasta estruturada.

1

Acesso corporativo

Confirmar acesso ao Claude Code com Fred Valente usando conta individual vinculada ao e-mail corporativo da Emotions/One Big Media. Nao compartilhar conta, sessao ou chave.

2

Instalacao do ambiente

Instalar VS Code, extensão Claude Code, Git, Node.js LTS, Python, Claude Code CLI e validar com claude doctor.

3

Primeira pasta estruturada

Criar ai-first-lab com docs, dados, scripts e app; gerar um projeto Next.js de treino para operar no VS Code.

4

Desafio inicial

Cada participante chega com 2 ou 3 dores reais da area para avaliacao de viabilidade, risco, impacto e dados disponiveis.

~/ai-first/agenda --semana-1..8

Lista do que vamos ensinar em cada encontro.

Cada encontro tem tres movimentos: conceito curto, pratica guiada e evolucao do MVP. A dor real de cada area e o fio condutor do curso.

01base

Fundamentos do Claude, Claude Code, VS Code e primeira pasta

Vamos ensinar
  • Diferença entre Claude App, chat, Claude Code CLI e extensão no VS Code.
  • Como abrir terminal integrado, navegar em pasta e usar comandos básicos.
  • Como estruturar um projeto para a IA entender contexto, decisões e dados.
Pratica do dia
  • Criar a pasta ai-first-lab no VS Code.
  • Criar README, docs/DESAFIO.md, docs/PROMPTS.md e docs/DECISOES.md.
  • Rodar Claude Code dentro da pasta e pedir explicação da estrutura.
Entrega
  • Ambiente validado no VS Code.
  • Primeira pasta estruturada.
  • Primeiro registro de decisão feito com apoio do Claude.
Base: projeto pronto para operar
02operar

Uso real do Claude: Plan Mode, prompt engineering e coworker

Vamos ensinar
  • Como pedir plano antes de editar arquivos.
  • Como escrever prompts com contexto, objetivo, restrições e formato.
  • Como revisar diff, comandos, arquivos alterados e próximos passos.
Pratica do dia
  • Usar Claude App/chat para pensar a tarefa.
  • Usar Claude Code no VS Code para planejar e executar uma mudança pequena.
  • Comparar um prompt ruim com um prompt bom.
Entrega
  • Biblioteca inicial de prompts em docs/PROMPTS.md.
  • Fluxo Plan → Execute → Review praticado.
  • Checklist pessoal de revisão humana.
Operação: Claude como coworker
03desafio

Transformar dores das áreas em escopo de MVP

Vamos ensinar
  • Como avaliar se um desafio cabe em 8 semanas.
  • Como mapear entrada, saída, regra, exceção e dono da validação.
  • Como classificar risco antes de automatizar.
Pratica do dia
  • Cada pessoa apresenta 1 a 3 dores da área.
  • O grupo prioriza por impacto, frequência, dados e segurança.
  • Claude ajuda a transformar a dor em escopo inicial.
Entrega
  • docs/DESAFIO.md preenchido.
  • Objetivo do MVP em uma frase.
  • Critérios de sucesso definidos.
MVP: problema validado
04dados

Dados, planilhas, arquivos e automações com Python

Vamos ensinar
  • Como ler CSV, Excel, TXT e pastas de arquivos.
  • Como limpar dados e padronizar colunas.
  • Como usar dados ficticios, mascarados ou minimizados.
Pratica do dia
  • Montar uma base de teste do desafio.
  • Criar pipeline simples de entrada e saida.
  • Validar resultado com casos reais anonimizados.
Entrega
  • Dados de teste preparados.
  • Script ou rotina de transformacao.
  • Regras de seguranca registradas.
MVP: pipeline de dados
05integrar

Primeira solução útil: app simples com Next.js

Vamos ensinar
  • Como criar uma tela simples em Next.js com ajuda do Claude.
  • Como separar input, processamento e resultado.
  • Como não transformar o app em caixa-preta.
Pratica do dia
  • Criar ou adaptar uma tela para o desafio.
  • Exibir resultado de uma planilha, arquivo ou script.
  • Revisar o código criado pelo Claude.
Entrega
  • Primeira interface funcional.
  • Fluxo manual substituído por tela ou comando simples.
  • README atualizado com como rodar.
MVP: primeira tela útil
06integrar

Integrações, APIs e ferramentas aprovadas

Vamos ensinar
  • O que é API, token, permissão e limite de acesso.
  • Quando usar planilha, arquivo, endpoint ou conector.
  • Como registrar nova ferramenta quando necessário.
Pratica do dia
  • Conectar o MVP a uma fonte aprovada.
  • Automatizar uma etapa manual da rotina.
  • Registrar riscos, acessos e dependências.
Entrega
  • Integração controlada funcionando.
  • Limites de permissão definidos.
  • Plano para remover acessos se o teste parar.
MVP: automação integrada
07seguro

Fechar MVP, testes, documentação e governança

Vamos ensinar
  • Como revisar segredos, dados e permissao.
  • Como documentar uso, limites e responsavel.
  • Como preparar uma solucao para avaliacao interna.
Pratica do dia
  • Revisar riscos do MVP.
  • Remover segredos, dados desnecessarios e acessos amplos.
  • Completar documentacao minima.
Entrega
  • MVP revisado com checklist de seguranca.
  • README de uso e manutencao.
  • Plano de continuidade ou descarte.

Demo Day: apresentacao, decisao e proximos passos

MVP: apresentado e avaliado
~/ai-first/desafios --por-area

Cada pessoa traz uma dor real da area.

A escolha do desafio e o ponto mais importante do programa. O melhor caso nao e o mais complexo: e aquele que tem impacto claro, dados acessiveis, risco controlado e alguem disposto a validar o resultado.

Como vamos avaliar cada desafio

  • Repeticao: acontece toda semana, todo mes ou consome muitas horas?
  • Impacto: reduz erro, economiza tempo, melhora receita, protege informacao ou acelera decisao?
  • Dados: os insumos existem e podem ser usados com seguranca?
  • Escopo: cabe em 8 semanas como MVP?
  • Risco: envolve producao, dados pessoais, financeiro, clientes, CMS, GCP, GitHub ou credenciais?
  • Dono: existe alguem da area que valide se a solucao realmente funciona?
Area e responsavelQuem traz o desafio e quem valida o resultado.
Processo atualComo e feito hoje, com quais ferramentas e quanto tempo consome.
Dor principalErro, retrabalho, atraso, custo, risco ou oportunidade perdida.
Dados envolvidosPlanilhas, arquivos, APIs, bancos, relatórios, e-mails ou sistemas.
Resultado esperadoO que precisa existir ao final para considerar o MVP util.
Limites de segurancaO que nao pode ser exposto, automatizado ou acessado sem aprovacao.
